= zlib bindings — first non-trivial example
:toc:

A worked example of `pascal_bindgen` against a real-world C header
(`/usr/include/zlib.h`, ~1.9 K LOC). Run `./generate.sh` from the
repo root to regenerate the two `.pas` files, then `./build.sh`
from this directory to build a small round-trip demo.

== What this example demonstrates

Two things at the same time:

1. *The pipeline is real* — `demo.pas` calls `zlibVersion`,
   `compressBound`, `compress`, and `uncompress` against the system
   `libz.so.1`, round-trips a 149-byte buffer down to 36 bytes and
   back, and verifies byte equality. The four `external`
   declarations were lifted verbatim from `zlib_fpc.pas` (parameter
   types only swapped from `^Bytef` to a named `PBytef` — see gap §5
   below); their `cdecl` convention, library name, and parameter
   shapes are exactly what the generator produced.
2. *The v1 emitter still cannot ship a usable `.pas` file for
   anything but the simplest header.* Compiling the full
   `zlib_fpc.pas` from this directory does not work: the gaps below
   compound. The demo is hand-trimmed evidence that the *kind* of
   declaration the emitter produces is correct, not that the *unit*
   it produces is consumable.

== Gaps surfaced by zlib

These are the v1 limitations zlib exposes, in roughly the order
they trip up `fpc -Cn examples/zlib/zlib_fpc.pas`:

=== 1. Typedefs from `#include`d headers are dropped (FIXED)

*Status: fixed (commit landed after the initial README write).*

The parser now uses `clang_Location_isInSystemHeader` as the
exclusion filter instead of `clang_Location_isFromMainFile`. Decls
that arrive via a user-side `#include` (zlib's `zconf.h`) now flow
through; decls from `<>`-style system includes (`<stddef.h>`,
`<stdio.h>`) still get dropped. The regenerated `zlib_fpc.pas`
contains the full vocabulary: `Bytef`, `uInt`, `uLong`, `voidpf`,
etc.

A few system-typedef use sites still leak via gap §4 — e.g.
`z_size_t = size_t;` references `size_t` which lives in `<stddef.h>`
and is therefore not declared. Resolving §4 (typedef-chain
unwrapping) maps these to canonical primitives at the use site.

=== 2. Function-pointer typedefs render to garbage (FIXED)

*Status: fixed.* The parser now detects `CXTypeKind_FunctionProto`
under any pointer or typedef and builds a `tkFunctionPointer` IR
node carrying the return type and ordered argument types. Both
emitters render Pascal procedural types:

[source,pascal]
----
alloc_func = function(arg1: Pointer; arg2: cuint; arg3: cuint): Pointer; cdecl;
free_func  = procedure(arg1: Pointer; arg2: Pointer); cdecl;
----

The same form appears inside `z_stream_s.zalloc` / `.zfree`,
`in_func` / `out_func`, and `inflateBack`'s parameters (which now
reference the typedef names directly instead of inlining the
proto). Blaise output is the same minus `cdecl` (Blaise hasn't
grown calling-conv syntax yet).

=== 3. Macros are silently skipped

zlib exports a few status codes (`Z_OK`, `Z_STREAM_END`, ...) and
flush constants (`Z_NO_FLUSH`, ...) as `#define` macros. The parser
runs with detailed preprocessing off and surfaces no `LossReason`
entries for these yet — the demo hard-codes `Z_OK = 0`.

Fix path: turn on `CXTranslationUnit_DetailedPreprocessingRecord`,
filter `MacroDef` cursors for integer-literal-only bodies, and emit
them as a `const` section alongside the enum constants.

=== 4. Typedef-chain unwrapping is one-shot (FIXED)

*Status: fixed.* `TBindingType` now carries a `CanonicalSpelling`
populated from libclang's canonical type. When a user typedef
references a system-header typedef that was filtered out
(`z_size_t = size_t` with `size_t` from `<stddef.h>`), the FPC
emitter falls back to `MapPrimitive(CanonicalSpelling)` and emits
`z_size_t = culong`.

=== 5. Anonymous pointer types in parameter lists (FIXED)

*Status: fixed.* The FPC emitter pre-walks every function in the
unit, collects pointee names that appear as a parameter or return
type, and emits a `P<X> = ^<X>` alias for each at the bottom of the
type section. Signatures then refer to the alias name. The
regenerated `zlib_fpc.pas` carries `PBytef`, `PuLongf`, `Pz_stream_s`,
`PuInt`, `PgzFile_s`, etc., and every pointer-parameter function
uses them in place of inline `^X`.

Inline `^X` in record fields and typedef RHS stays — FPC accepts it
there, so we do not invent unnecessary aliases.

=== 6. Duplicate type declarations (FIXED)

*Status: fixed.* The emitters maintain a per-`Emit()` set of
already-written type names; `EmitDecl` short-circuits on dupes.
`gzFile_s` now appears exactly once in `zlib_fpc.pas`.

=== 7. Reserved-word collisions (FIXED for FPC)

*Status: fixed for the FPC emitter; Blaise side deferred (Phase C).*
The FPC emitter wraps every emitted identifier through an
`EscapeIdent` helper that prepends `&` when the name hits a 68-word
reserved table. The zlib output now shows `&in`, `&out`, `&file`
where C-side parameters collide.

== Reproducing

[source,sh]
----
# regenerate the two .pas files (run from repo root)
./examples/zlib/generate.sh

# build the trimmed demo against libz (run from examples/zlib/)
./build.sh
./demo
----

Expected output:

----
zlib version: 1.3
compressed 149 -> 36 bytes
round-trip OK
----

== Phase A / B status

Phases A and B (gaps §1, §2, §4, §5, §6, §7) are complete. The
regenerated `zlib_fpc.pas` now `fpc -Cn`s with a single remaining
error: `gzvprintf` takes a `va_list` parameter (`struct
__va_list_tag[1]` after glibc expansion). That's a platform-ABI
construct outside the IR's current vocabulary; the variadic
counterpart `gzprintf` works via `varargs` already. Skipping
`va_list`-typed decls is a tiny follow-up.

Gap §3 (macro constants) is still open and is the remaining
Phase B item.

The sample header in `src/test/resources/sample.h` was carefully
constructed to avoid every one of these, which is why the existing
test suite stayed green throughout — and the new regression tests
in `sample_with_include.h` / `sample_dup.h` / `sample_reserved.h` /
`sample_ptr.h` exercise each Phase A fix in isolation.
