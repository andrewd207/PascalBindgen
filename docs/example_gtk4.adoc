= Worked example — GTK 4 bindings
:toc:

Start-to-finish walkthrough: build the tool, point it at a system
header, generate Pascal bindings, and link a working program against
them. We use GTK 4 because it exercises a realistic chunk of the
generator (large `#include` chain, opaque types, function pointers,
enums, varargs, calling conventions) and the result is something you
can see on screen.

This example produces FPC output. The Blaise and rqbasic paths are
identical except for the dialect flag and the file extension.

== 0. Prerequisites

Debian / Ubuntu names; adapt for your distro.

[source,shell]
----
# Build-time
sudo apt install fpc pasbuild libclang-17-dev clang-17

# GTK 4 headers + runtime
sudo apt install libgtk-4-dev
----

The `libclang-17` runtime is what `libclang_shim.so` loads at run
time. The version on disk and the SONAME baked into the shim must
match; rebuild the shim against your local libclang if you have a
different major version.

== 1. Build pascal_bindgen

[source,shell]
----
cd pascal_bindgen
pasbuild
----

This compiles `src/main/c/libclang_shim.c` → `src/main/c/libclang_shim.so`
(via the build hook) and `src/main/pascal/Main.pas` →
`target/pascal_bindgen`. The binary's rpath points back at the shim,
so it runs in place without `LD_LIBRARY_PATH`.

Smoke test:

[source,shell]
----
./target/pascal_bindgen --header /usr/include/stdio.h
----

Should print a one-line-per-decl dump on stdout. If you see clang
errors instead, your `libclang_shim.so` can't find its libclang —
re-check the version pin.

== 2. Discover the GTK 4 include paths

GTK 4 lives in a constellation of include directories (GLib,
GObject, Pango, Cairo, …​). Don't write them by hand; ask
`pkg-config`:

[source,shell]
----
pkg-config --cflags gtk4
----

Sample output:

----
-I/usr/include/gtk-4.0 -I/usr/include/glib-2.0
-I/usr/lib/x86_64-linux-gnu/glib-2.0/include -I/usr/include/pango-1.0
-I/usr/include/cairo -I/usr/include/gdk-pixbuf-2.0
-I/usr/include/graphene-1.0 ...
----

pascal_bindgen forwards anything after `--` verbatim to clang, so we
will just splice that command into the generate call.

== 3. Generate the bindings (FPC dialect)

[source,shell]
----
./target/pascal_bindgen --fpc \
    --header /usr/include/gtk-4.0/gtk/gtk.h \
    --output gtk4_fpc.pas \
    --unit-name gtk4_fpc \
    --library libgtk-4 \
    -- $(pkg-config --cflags gtk4)
----

What each flag does:

`--fpc`::
  Pick the Free Pascal emitter.

`--header`::
  Single root header. clang follows its `#include` chain; everything
  the chain visits that *isn't* a system header is in scope. (System
  headers like `stddef.h` contribute typedef vocabulary but no
  decls — see `bindgen.parser.pas` for the filter.)

`--output gtk4_fpc.pas`::
  Destination file. Use `-` (or omit) to write to stdout.

`--unit-name gtk4_fpc`::
  Goes into `unit gtk4_fpc;` at the top. Optional — defaults to the
  output file basename.

`--library libgtk-4`::
  Attached to every emitted `external` directive as
  `external 'libgtk-4' name 'gtk_window_new';`. The runtime loader
  needs to be able to find `libgtk-4.so`; we deal with that at link
  time in the next step.

`-- $(pkg-config --cflags gtk4)`::
  Pass-through clang args. Without these clang would not be able to
  find `glib.h`, `cairo.h`, etc., and parsing would fail.

The generated file starts with a provenance comment:

[source,pascal]
----
{ DO NOT EDIT — generated by pascal_bindgen.
  source: /usr/include/gtk-4.0/gtk/gtk.h
}

unit gtk4_fpc;

{$mode objfpc}{$H+}
{$PACKRECORDS C}

interface

uses
  ctypes;

type
  ...
----

== 4. Write a tiny demo

`demo.pas`:

[source,pascal]
----
program demo;

{$mode objfpc}{$H+}

uses
  ctypes, gtk4_fpc;

procedure on_activate(app: PGtkApplication; user_data: gpointer); cdecl;
var
  win: PGtkWidget;
begin
  win := gtk_application_window_new(app);
  gtk_window_set_title(PGtkWindow(win), 'Hello from pascal_bindgen');
  gtk_window_set_default_size(PGtkWindow(win), 400, 200);
  gtk_window_present(PGtkWindow(win));
end;

var
  app: PGtkApplication;
  status: cint;
begin
  app := gtk_application_new('org.example.hello', G_APPLICATION_DEFAULT_FLAGS);
  g_signal_connect_data(app, 'activate',
      GCallback(@on_activate), nil, nil, 0);
  status := g_application_run(GApplication(app), 0, nil);
  g_object_unref(app);
  Halt(status);
end.
----

== 5. Link against the real GTK 4 SONAMEs

The `external 'libgtk-4'` directive looks for `libgtk-4.so` — a
symlink that Debian only ships in the `-dev` package. To avoid that
dependency at run time, pre-load the actual SONAMEs from the linker
command line. The plain symbol resolution still works because the
symbols are already in the process by the time the dynamic loader
checks.

[source,shell]
----
LIBDIR=/usr/lib/x86_64-linux-gnu

fpc -Mobjfpc -O1 \
    -k"-rpath=$LIBDIR" \
    -k"$LIBDIR/libgtk-4.so.1" \
    -k"$LIBDIR/libgio-2.0.so.0" \
    -k"$LIBDIR/libgobject-2.0.so.0" \
    -k"$LIBDIR/libglib-2.0.so.0" \
    demo.pas
----

Then:

[source,shell]
----
./demo
----

A GTK 4 window titled "Hello from pascal_bindgen" should open.

== 6. Same headers, different dialects

The same generate command with a different dialect flag produces
output for the other compilers.

Blaise (`.pas` consumed by the Blaise compiler):

[source,shell]
----
./target/pascal_bindgen --blaise \
    --header /usr/include/gtk-4.0/gtk/gtk.h \
    --output gtk4_blaise.pas \
    --unit-name gtk4_blaise \
    --library libgtk-4 \
    -- $(pkg-config --cflags gtk4)
----

RapidQ-ll (`.bas` consumed by `rapidq`):

[source,shell]
----
./target/pascal_bindgen --rqbasic \
    --header /usr/include/gtk-4.0/gtk/gtk.h \
    --output gtk4_rqbasic.bas \
    --library libgtk-4 \
    --prefix-types \
    -- $(pkg-config --cflags gtk4)
----

Sample demos for each dialect are checked into `examples/gtk4/`
(FPC), `examples/blaise-gtk4/` (Blaise), and
`examples/gtk4/demo_rqbasic.bas` (rqbasic), along with build
scripts.

== 7. Regenerating

Generated `.pas` / `.bas` files are output, not source of truth.
When the header changes (new GTK release, different `-D` switches,
etc.) re-run the generate command. The provenance comment at the
top tells you the exact input the file was built from.

A common pattern is to wrap the generate call in a shell script
beside the bindings, so the regeneration recipe is in the same
folder as its output. `examples/gtk4/` follows this layout.
