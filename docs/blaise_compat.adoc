= Blaise compatibility notes
:toc:

Issues hit while porting `pascal_bindgen`'s own source to compile under
both FPC and the Blaise compiler. Reported with minimal repros to make
fixes upstream straightforward. Blaise is a young Pascal compiler under
active development; everything here is expected to improve.

Tested against:

* FPC 3.3.1 (Linux x86_64, 2026-05-12 build)
* Blaise v0.9.0-dev via `compiler/bootstrap/blaise_unit_source`
* Blaise v0.10.0-SNAPSHOT (re-check of the FItems[N] := X issue only)
* Blaise v0.11.0-dev (`/tmp/blaise-master`, commit `40755b9`) — full sweep
* Dates: 2026-05-27 (initial), 2026-06-02 (re-check), 2026-06-09 (v0.11 sweep)

== 2026-06-09 v0.11.0-dev re-check summary

Newly fixed (now compile clean):

* `const` parameter qualifier
* `@OutOrVarParam` (address of `var`/`out` parameter)
* `@dynArr[0]` (address of dyn-array element)
* `Exception.CreateFmt(fmt, [...])`

Still broken (unchanged from prior runs):

* Forward class declaration (`TFoo = class;`)
* `default;` on indexed property
* Bare `inherited;`
* `var X: T = init` initializer at declaration
* `arr[I].MethodCall` (must stage through a local)
* `class function ... static`
* `FItems[N] := X` (class-field dynamic-array indexed write)
* `P[i]` indexing on typed pointer (`PByte = ^Byte`)
* Constant-expression array bounds (`array[0..CHUNK-1]`)

== By-value record ABI (SysV) — 2026-06-09

Recent Blaise work added by-value record params/returns (commits
`9ed5679`, `18e5826`, et al.). Tested through the **QBE backend**
(not the `--backend native` path — that one is broken even for
params on this case):

* **Pass POD record by value**: **works**. For
  `CXString { PtrUInt data; Cardinal private_flags; }` (16 bytes,
  no ARC fields), QBE splits the value across `RDI:RSI` per SysV.
  `clang_getCString(s)` with a hand-built `s` returns the right
  bytes.
* **Receive POD record returned by value**: **broken**. Blaise's IR
  emits the call as `call $f(l $s)` — sret-via-hidden-pointer — for
  the return path, even when the record has no ARC fields and would
  classify as INTEGER+INTEGER (returned in `RAX:RDX`) under SysV.
  `s := clang_getClangVersion();` leaves both fields zero because
  the real libclang function doesn't take a hidden sret arg.
  Declaring the function as `procedure (out s: CXString)` doesn't
  rescue it — the call shape Blaise emits still mismatches the C
  ABI.

Net effect for this project: pascal_bindgen itself only needs to
build under FPC — it just emits Blaise units, it doesn't need to be
hosted on Blaise. So the libclang shim has been removed (2026-06-10)
and `clang.ffi.pas` now declares libclang's by-value entry points
directly. The Blaise codegen path through this FFI is verified
working on a smaller harness (`t_pass.pas` / `t_sysv2.pas` round-
trip CXString via the LLVM backend), so once Blaise's published
bootstrap binaries catch up with the current stdlib (specifically:
parse `Exit(value)` and expose `--backend llvm` in the same build),
`tools/blaise_link.py` can compile pascal_bindgen end-to-end with
no further changes here.

== Compiler driver / link surface

[cols="1,2,2"]
|===
| Concern | FPC | Blaise

| Library link via directive    | `{$linklib X}` honored        | Directive silently consumed (`uLexer.pas:472`); no `-l` flag emitted
| Library link via CLI / env    | `-Fl<dir>`, `-k-l<name>`      | None — `cc` arg list is hardcoded in `Blaise.pas:511-527`
| Loader-resolved `external 'lib'` | `external 'libfoo'` syntax  | No equivalent; externs are bare `external name 'sym'`
| Calling conventions in extern | `cdecl`/`stdcall`/...         | None — externs carry only `name '...'`
| Varargs                        | `varargs;` modifier           | None
|===

Workaround in this project: `tools/blaise_link.py` runs `blaise
--emit-ir`, then drives `clang -c` + `cc` manually with the shim
linked in.

== Parser / syntactic

[cols="1,2,2"]
|===
| Concern | FPC | Blaise

| Forward class declaration            | `TFoo = class;`                       | Not supported in any spelling (`class;`, `class forward;`, `class; forward;` all rejected) — see comment in `blaise.testing.pas`
| `default;` on indexed property       | Supported, enables `someList[I]`      | Rejected (parse error after `default`); call sites must use `someList.Items[I]`
| `inherited;` bare                    | Resolves by name to parent method     | Requires explicit method name: `inherited Destroy;` / `inherited Create;`
| `{$mode}` / `{$H+}` directives       | Required for modern features          | Unknown directives are silently consumed; gate FPC-mode-dependent code with `{$IFDEF FPC} ... {$ENDIF}`
| `{$modeswitch advancedrecords}`      | Required for records-with-methods     | Records with methods accepted natively
| `var X: T = default` initializer     | Supported                             | Parse error — initialize in `begin` body instead
| `arr[I].MethodCall`                  | OK                                    | Parse error; assign to a local first: `obj := arr[I]; obj.Method;`
| `class function` / `class procedure` | OK with `static;` qualifier           | Not yet parsed (`Expected 'end' in class definition`); use a plain unit-level function
| `const` parameter qualifier          | Supported                             | **Supported as of v0.11.0-dev** (2026-06-09)
| `FItems[N] := Item;` (writing to array-of-T class field) | OK | Semantic error "Undeclared variable 'FItems'" — see repro below. **Still reproduces on v0.10.0-SNAPSHOT (2026-06-02)** even though local-`var` dynamic arrays now work end-to-end (`SetLength` + indexed read/write fine). Only the class-field indexed-write path is broken.
| `Format(fmt, [...])` (SysUtils) | Supported via `TVarRec` | **Supported** — handles mixed Integer/string args, e.g. `Format('rc=%d for "%s"', [rc, fn])` works.
| `Exception.CreateFmt(fmt, [...])` | Supported | **Supported as of v0.11.0-dev** (2026-06-09).
| `@OutOrVarParam` (taking address of a `var`/`out` parameter) | OK | **Supported as of v0.11.0-dev** (2026-06-09).
| `@dynArr[0]` (address of dyn-array element)               | OK | **Supported as of v0.11.0-dev** (2026-06-09).
| `P[i]` indexing of typed pointer (`PByte = ^Byte`)        | OK in mode delphi | `Semantic error: 'P' is not a static array or dynamic array` (or, with a cast, `String subscript '[]' requires a string expression`). The `type PByte = ^Byte` declaration itself parses, and typed-pointer arithmetic + deref works (`(P + i)^`); only `[]` dispatch on a pointer is missing.
| Constant-expression array bounds (e.g. `array[0..CHUNK-1]`) | OK     | `Parse error: Expected integer bound`. Workaround: literal bound.
|===

.Minimal repro for the array-field-assignment issue:
[source,pascal]
----
program aw;
type
  TFoo = class
    FItems: array of Integer;
    procedure Add(X: Integer);
  end;
procedure TFoo.Add(X: Integer);
var N: Integer;
begin
  N := Length(FItems);        // OK
  SetLength(FItems, N + 1);   // OK
  FItems[N] := X;             // Semantic error: Undeclared variable 'FItems'
end;
begin end.
----

Reading from an array-of-T field works (`Result := FItems[I]`), as does
passing the field to `Length`/`SetLength`. Only the indexed-write form
trips the semantic check. Workaround in this project: pending.

== Standard library

[cols="1,2,2"]
|===
| Concern | FPC | Blaise

| `ctypes` unit (`cint`, `cint64`, ...) | Available                       | Absent — use built-ins (`Integer`, `Int64`, `Cardinal`, ...)
| `fgl` (`TFPGObjectList<T>`)            | Available                      | Absent — Blaise has `Generics.Collections.TList<T>` (no `TObjectList<T>` yet)
| `Classes.TList` (untyped pointer list) | Available                      | Not in `blaise.stdlib.classes.pas`
| `PAnsiChar`                            | Available                      | Absent — use `PChar` (still UTF-8 byte pointer)
| `Char` / `AnsiChar` types              | Available                      | No `Char` type — strings are UTF-8; `str[i]` returns `Integer`
| `fpcunit` / `consoletestrunner`        | Available                      | Replaced by `blaise.testing` + `blaise.testing.runner.text` (same xUnit API surface)
| `Move` (RTL bulk memcopy)              | Available                      | Absent — bind `memcpy` directly via `procedure _Memcpy(Dst, Src: Pointer; N: Int64); external name 'memcpy';`
| `ReallocMem(P, N)`                     | Procedure, mutates `P` via var | Function returning the new pointer — `P := ReallocMem(P, N);` (gated by `{$IFDEF FPC}`)
| `File` (untyped file) + `AssignFile` / `BlockWrite` / `BlockRead` | Available | Absent — use `streams.TFileInputStream` / `TFileOutputStream`
| `PByte` (RTL pointer alias)            | Available                      | Absent — `type PByte = ^Byte;` declares fine, but see indexing row above
|===

== Records / layout

Blaise uses natural alignment that matches the C ABI by default — no
`{$PACKRECORDS C}`-equivalent needed in emitted units. Byte=1, UInt16=2,
Integer=4, Int64=8.

Variant records (`record case Integer of ...`) are not supported. The
Blaise emitter represents C unions as a record holding only the first
alternative + a loss comment.

== Out of scope (for this doc)

Behavioural differences that don't block dual-build:

* String indexing is 0-based in Blaise vs 1-based in FPC (affects
  per-char code; we avoid both).
* Long double width (Blaise has no 80-bit float; emitter maps `long
  double` → `Double`).
